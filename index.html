ℹPassword is consumed by Firebase only. It is never sent to Django and never stored in localStorage.
⚠
ToS & Privacy Policy consent is collected on this screen via a required checkbox. Three values must be recorded:

tos_agreed — boolean, must be true to proceed
tos_agreed_at — ISO 8601 timestamp, set client-side at moment of checkbox tick
tos_agreed_ip — see open question below
!
Open question — IP address capture: Two options:

Option A (recommended): Django reads REMOTE_ADDR / X-Forwarded-For from the request server-side. More reliable, cannot be spoofed by the client. No extra frontend work.

Option B: Frontend fetches the user's IP (e.g. via api.ipify.org) and sends it as tos_agreed_ip in the request body. Simpler but the value can be manipulated.

Team needs to decide before implementation. Option A is strongly preferred for legal/compliance purposes.

Stripe → Django: POST /stripe/webhook/ (server-side)

Stripe fires this independently of the browser after payment is confirmed.

// Stripe webhook payload
{
  "type": "checkout.session.completed",
  "data": {
    "object": {
      "id":             "cs_live_a1b2c3d4e5",
      "payment_status": "paid",
      "subscription":   "sub_1NkQzBPwXy9876",
      "customer":       "cus_Abc123"
    }
  }
}
// Django responds 200 OK
⚠Webhook fires asynchronously. The browser redirect back to the app can arrive before the webhook — frontend must not depend on webhook completing before the return URL is processed.
Stripe → Frontend: Return URL Handling

Stripe redirects browser back to the onboarding URL with query params.

// Success return
?payment_success=true&session_id=cs_live_a1b2...

// Detected in RegistrationFlow.tsx
if (searchParams.get("payment_success")) {
  localStorage.payment_status = "paid"
  localStorage.session_id     = session_id
  completeStep("subscription")
}

// Cancellation return
?payment_cancelled=true

if (searchParams.get("payment_cancelled")) {
  // show: "Payment was not completed.
  //        Please try again."
  // stay on Subscription step
}
localStorage ← payment_status: "paid"  |  session_id: "cs_live_a1b2..."

Stripe → Frontend: Return URL Handling

Stripe redirects back with one of two params depending on outcome.

// Happy path: onboarding completed
?connect_return=true

if (searchParams.get("connect_return")) {
  localStorage.stripe_account_id    = "acct_1NkQzBPwXy9876"
  localStorage.stripe_connect_status = "active"
  completeStep("commerce")
}

// Expired link: merchant took too long
?connect_refresh=true

if (searchParams.get("connect_refresh")) {
  // Call GET /stripe/onboarding-link/ again
  // Redirect to the new URL
  // Do NOT show error to merchant
}
localStorage ← stripe_account_id: "acct_1Nk..."  |  stripe_connect_status: "active"
⚠connect_refresh means the Stripe link expired (15–30 min timeout). Silently re-fetch and redirect — do NOT show an error message. The merchant will see the same Stripe screen again.
